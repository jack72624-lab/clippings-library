一句話

這篇不是知識增量，是官方幫你核對現有做法方向對不對；真正能再拿走的，只有那三個操作切片。

Boris Cherny 是 Claude Code 的創建者，這場官方演講把「新手第一天到熟練用戶」的一整條路徑講了一遍：從問 codebase 問題上手，到寫 code 前先規劃，到串接團隊的 CLI／MCP 工具，到給它一個回饋管道讓它自動迭代，到用 CLAUDE.md 階層灌上下文，再到一排 CLI 快捷鍵。電腦王阿達做了中文搬運（逐字稿裡有幾處語音辨識把專有名詞聽錯，例如 Claude 被聽成 quad、slash command 被聽成 splash command，這裡引用時都已經校正過）。

但對你這種已經把 Claude Code 用到這個深度的人，這篇的意義不是學新東西，而是拿官方基準線對照自己的 setup：七項技巧，你落在哪一格。

01七大技巧，你落在哪▶ 4:06

七項裡，六項你已經做到甚至做得更深，只剩一項還算大致熟，沒有一項是真的缺口。已做到的兩項最基本：Boris 說新人第一天先問 codebase 問題（不索引、code 留本機、不上傳）能把 onboarding 從兩三週縮到兩三天，你雖然是 solo、沒有「帶新人」的情境，但接手各專案時本來就是靠讀 CLAUDE.md 加探索上手；一句「commit push」讓它自己看 git log 推敲格式、開 PR，你的 git push／gcloud builds／firebase hosting 也早就設放行、日常在用。

已超越的四項，才是這篇真正的印證：官方說「寫 code 前先 brainstorm、plan」，你的 CLAUDE.md 第一條規則更嚴，要等明確確認才動手，未確認不得寫檔、跑 shell；官方說把團隊 CLI 和 MCP 工具都給它用，你接的是一整排 firecrawl、playwright、notebooklm、context7、twinkle-hub、Gmail、Calendar、Drive，還有一堆自建 skills；官方說「給它一個能檢查自己成果的管道，它就會自動迭代」，你走的是 Playwright CDP 截圖驗證加 wireframe-first（假資料真邏輯的互動線框先讓你試用定案，再翻正式版對帳），比影片講的更成熟；官方講 CLAUDE.md 五層階層，你除了這五層，還多了 auto-memory 加 CHANGELOG 三檔分工，外加多視窗同步的護欄機制。

官方公開技巧兜了一輪，你花最多力氣建的三根支柱全被印證：先規劃、CLAUDE.md 階層、給回饋自動迭代。

02真正的淨增益：三個操作切片▶ 20:44

對照表之外，還有三個 Boris 帶到、但你 setup 裡「還沒有／可以再拉一階」的操作。按增益大小排：

第一個切片最值得動手。Boris 把 claude -p 形容成「一個超智慧的 unix utility」：給它 prompt、回你 JSON，可以任意串接，Anthropic 內部用它跑 CI、事件回應、各種 pipeline。用法很直接，claude -p "prompt" --allowed-tools ... --output-format json（或 stream-json 邊跑邊處理），可以 pipe in／pipe out，例如 git status | claude -p "..."，或讀一整個 GCP bucket 的巨大 log 導進去叫它「找出這裡有趣的地方」，輸出再用 jq 篩。你的訊息 bridge 已經在用它的唯讀模式回覆非 {} 格式訊息，但 Boris 講的是把它當管線元件串進更多自動化，剛好對得上你那一排 launchd job：如果那些 job 現在還在用完整互動式 Claude 或土法 script，改成 claude -p 管線會更穩、更好接。

第二個切片是 git worktree。Boris 說重度用戶常同時開多個 Claude，同一 repo 開多個 checkout 或用 worktree 隔離，讓多個 Claude 平行跑同一 repo 不互相打架。你的分視窗平行工作法是手動開多個視窗、主框當控制塔，靠 sync-guard hook 擋「同一份檔被兩個視窗同時改」的風險；git worktree 是它的升級版：每個平行 Claude 在自己的工作目錄，共用同一個 .git，改檔天生不會互相覆蓋。順帶一提，這個 harness 的 subagent isolation: "worktree" 就是同一個概念的內建版。

第三個切片增益最小：macOS 系統設定裡開啟聽寫，連按兩下聽寫鍵就能直接對 Claude Code 口說 prompt。純習慣問題，跟訊息 bridge 不衝突，想在電腦前口述長指令時可以試試。

03番外：兩個順口透露的觀點▶ 23:51

技巧之外，Boris 在演講裡順口講了兩件事，比技巧本身更值得記一筆。

「今年底大家可能就不用 IDE 了。」

他解釋為什麼做 CLI 而不做 IDE 外掛：一是 Anthropic 內部 IDE 太雜，VS Code、Zed、Xcode、Vim、Emacs 都有人用，終端機是最大公約數；二是他們近距離看到模型進步速度太快，刻意不過度投資 UI 上層，因為很快可能就用不上。（影片是 2026 年 4 月錄的，這是他當時的預測，當一個時間戳看就好。）另一個數字：Anthropic 約 80% 技術人員每天用 Claude Code，含研究員（用它編輯、執行 notebook），這是dogfooding做到什麼程度的一個具體數字。

呼應你的踩坑

Q&A 環節有人問最難做的部分是什麼，Boris 說是讓 bash 指令安全：先用靜態分析判斷哪些指令是唯讀、能安全組合，再疊一套多層級權限系統，各層各自設 allow／block 清單。這正是你常遇到「某些指令被分類器擋」背後的機制，不是 bug，是設計。

04兩張心法圖，把邏輯串起來▶ 12:08

對照表和切片之外，Boris 講的兩個機制本身值得單獨畫出來看。

—怎麼用在你身上

  • launchd job 改走管線：morning briefing、週摘要、知識庫 Lint、clippings 提醒這排排程，逐個檢查是不是還在用完整互動式 Claude，能改就改成 claude -p + --output-format json + jq。
  • 下次多視窗平行改同一個 repo：改用 git worktree add 開隔離分身，比多視窗共用工作目錄乾淨，也少一層 sync-guard 要擋的風險。
  • 被分類器擋的時候別煩躁：那是「唯讀判斷 + 多層權限」的設計結果，不是系統壞了，換一條唯讀 API 互補路徑或用 ! 自跑指令繞過就好。
