一句話

這不是私人聊天機器人的進化，是把 Claude 變成組織級、有身分、有工具、能被信任動手的團隊成員。

Andrej Karpathy 把這件事放進一個更大的框架裡：LLM 介面正在經歷第三次重新設計。第一次是你會主動去的「網站」，第二次是你下載到手機裡的「App」，這一次，是一個自帶脈絡、常駐、非同步、擁有組織級工具、跟人類團隊並肩工作的實體。Claude Tag，就是這個說法目前最具體的實證。▶ 12:55

01從「問答框」到「有帳號的隊友」▶ 9:51

官方的定義很直白：Claude Tag 是「Claude Code 的進化，更主動、為整個團隊設計」。你在 Slack 裡 @ 它一下，它就把任務拆成幾個階段，動用手上被授權的工具把事情做完，再回到討論串貼出成果，不用你逐步交代怎麼做。

更關鍵的是脈絡的歸屬變了。過去每個人開自己的對話框，脈絡只留在那個視窗裡；現在是一個頻道對應一個共享的 Claude，它跟著頻道一起累積脈絡，隊友能直接接手你昨天交辦到一半的事，不用每次都從頭交代。開啟ambient 模式之後，它甚至會主動跟進那些冷掉沒人回的討論串，跨頻道標出彼此相關的事。

02它真的會動手，不只是回答▶ 13:44

一個實測者的說法很具體：「丟一個 codebase 給它，它讀完列出 10 個最該修的問題、排序、改好、開 review，全程都在 Slack 裡完成。」寫 PR、合併 PR、跑資料分析、處理 incident、做 bug triage、盯 metrics：這些原本要人手動切換好幾個工具才能做的事，現在是 @ 一下、等它回報。

一句話帶走

Anthropic 產品團隊現在65% 的代碼來自內部版的 Claude Tag，兩篇不同來源的報導都給了同一個數字。

03權限怎麼切，才敢讓它真的動手▶ 14:56

能放心讓一個 AI 隊友動手的前提，是權限要切乾淨。做法是按頻道隔離：法務群裡的那個分身看得到合約細節，但碰不到研發頻道的代碼；反過來也一樣，越權會直接被拒絕：不是靠它「自律」，是它手上的那把鑰匙本來就開不了另一扇門。所有操作也都記在它自己的帳號日誌裡，可回溯、可追責。

Tariq 的設定撇步值得記下來：把 pin message 加指令，當作那個頻道的「CLAUDE.md」（會被它記住）；另外開一個像 你的名字-Claude 的個人頻道，放自己的工作，把訊息轉進去交辦。

04別只看炒作：它的挑戰▶ 19:59

完整面：不只是好處

「many Claudes 很混亂」：Slack 裡沒有單一一個「我的 Claude」，每個頻道各一個、由 admin 設定，都不真正屬於你。

人的層面摩擦：你一 tag 它，它就讀了所有人的訊息，容易變成「你帶進團隊工作區的監控裝置」，做得好被酸「外包思考」，做不好被酸「早說了會這樣」，兩面都不討好。

設定比看起來複雜：access／工具設定一堆眉角，Anthropic 得另外出一份 agent identity 與 permissions 的最佳實踐文件。

鎖定疑慮：尤其在 Fable 事件之後，有團隊寧可自建 Slack agent，也不想把團隊的協作核心綁在單一封閉供應商上。

從 prompting 到 delegation：不再是每一步都要交代，而是給一個目標，讓它自己拆解、動手、回報。

—怎麼用在你身上

  • 你早就手刻過個人版：你的訊息 bridge 本質上就是「把 Claude 放進日常通訊軟體、常駐、用大白話交辦」，Claude Tag 是這套的官方產品化＋多人版，可以回頭拿官方怎麼解 resume／記憶／身分，對照自己 bridge 的設計。
  • prompting → delegation，正是 plan-pipeline 和分視窗平行工作法的核心：給目標，不給逐步指令。
  • 「pin message 當頻道 CLAUDE.md」：這套機制可以直接遷移到任何要接的 agent 上，用一則置頂訊息當作那個場域的「守則」。
